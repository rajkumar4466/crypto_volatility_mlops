Create `src/monitoring/drift.py` with a single public function `compute_drift(reference_df, recent_df, features, p_threshold=0.01, min_drifted=2)`:
- Accepts two pandas DataFrames (reference window = training data loaded from S3 Parquet; recent window = last N rows from Feast offline store or DynamoDB)
- Iterates over the `features` list (all 12 feature names: volatility_10m, volatility_30m, volatility_ratio, rsi_14, volume_spike, volume_trend, price_range_30m, sma_10_vs_sma_30, max_drawdown_30m, candle_body_avg, hour_of_day, day_of_week)
- For each feature calls `scipy.stats.ks_2samp(reference_df[feat].dropna().values, recent_df[feat].dropna().values)`
- Collects features where pvalue < p_threshold into a `drifted` list
- Returns tuple: `(drift_detected: bool, drift_score: float, drifted_features: list[str])`
  - `drift_detected = len(drifted) >= min_drifted`
  - `drift_score = len(drifted) / len(features)` (fraction 0.0–1.0)
- Skip KS-test for any feature with fewer than 30 non-null samples (insufficient data); log a warning but do not count as drift
- CRITICAL: Do NOT trigger drift on a single feature. The 2-of-N gate is enforced by `min_drifted=2`, which is the project's confirmed threshold (STATE.md).

Create `src/monitoring/accuracy.py` with a single public function `compute_rolling_accuracy(table_name: str, window_minutes: int = 120) -> float | None`:
- Use `boto3.resource('dynamodb').Table(table_name).scan()` with FilterExpression
- FilterExpression: `#ts >= :cutoff AND attribute_exists(actual_label)` where cutoff is `(datetime.utcnow() - timedelta(minutes=window_minutes)).isoformat()`
- ExpressionAttributeNames: `{"#ts": "timestamp"}` (avoids reserved word conflict)
- Handle DynamoDB paginated scan: if `response.get("LastEvaluatedKey")`, continue scanning with `ExclusiveStartKey`
- From returned items: compare `item["prediction"] == item["actual_label"]` for accuracy
- Return `None` if fewer than 10 items found (not enough data for reliable accuracy estimate)
- Return `float` accuracy in range [0.0, 1.0] otherwise
- Do NOT raise exceptions on empty table — return None cleanly (first DAG cycles before backfill exists)
  </action>

  ```python
  Namespace='CryptoVolatility/Monitoring'
  MetricData=[
    {'MetricName': 'rolling_accuracy', 'Value': rolling_accuracy, 'Unit': 'None', 'Timestamp': now},
    {'MetricName': 'drift_score', 'Value': drift_score, 'Unit': 'None', 'Timestamp': now},

  ]
  ```
- Use `datetime.utcnow()` for Timestamp (CloudWatch requires within 2 weeks past / 2 hours future)
- Log which metrics were skipped (None) at DEBUG level
- Do NOT create CloudWatch alarms in Python — alarms are Terraform resources (Plan 06-02)
- Do NOT publish more than these 5 metrics — CloudWatch free tier is 10 custom metrics

Create `src/monitoring/retrain_trigger.py` with a single public function `trigger_retrain_dag(airflow_host: str, dag_id: str, jwt_token: str) -> dict`:
- POST to `{airflow_host}/api/v2/dags/{dag_id}/dagRuns`
- Headers: `{"Authorization": f"Bearer {jwt_token}", "Content-Type": "application/json"}`
- Body: `{"dag_run_id": f"drift_{datetime.utcnow().strftime('%Y%m%dT%H%M%SZ')}", "conf": {"trigger_reason": "feature_drift"}}`
  - The unique `dag_run_id` prevents duplicate DAG runs if monitor task retries
- Set `timeout=30` seconds
- Call `response.raise_for_status()` — let Airflow HTTP errors propagate clearly
- Return `response.json()` on success
- CRITICAL: Endpoint is `/api/v2/dags/...` NOT `/api/v1/dags/...`. Airflow 3.x removed `/api/v1` entirely.
- Wrap in try/except for `requests.exceptions.HTTPError`: if status code is 409 (Conflict — DAG run with that ID already exists), log a warning and return gracefully without re-raising (idempotent behavior for Airflow retries)

The monitor task must:
1. Load reference features from S3 (`s3://S3_BUCKET/features/reference/reference_features.parquet`) using `pandas.read_parquet` with `s3fs` or boto3 presigned URL. If file does not exist (first run), skip drift detection and log a warning.
2. Load recent features from Feast offline store (last 60 rows from the S3 Parquet offline store path, sorted by event_timestamp descending). Alternatively, load from DynamoDB prediction log's `features` field. Use whichever approach is consistent with how Phase 2 wrote features.
3. Call `compute_drift(reference_df, recent_df, FEATURE_NAMES)` where `FEATURE_NAMES` is the list of all 12 features.
4. Call `compute_rolling_accuracy(os.environ['PREDICTIONS_TABLE_NAME'])`.
5. Read `model_version` from S3 (`s3://S3_BUCKET/models/current_metrics.json`, key `version`).
6. Read `prediction_latency` from CloudWatch (use `get_metric_statistics` on namespace `AWS/Lambda`, metric `Duration`, last 5 minutes). If unavailable, pass `None`.
7. Compute `retrain_count` as 0 for now (retrain tracking can be enhanced post-Phase 6).
8. Call `publish_metrics(rolling_accuracy, drift_score, model_version, prediction_latency, retrain_count)`.
9. If `drift_detected=True` AND `os.environ.get('AIRFLOW_API_TOKEN')` is set, call `trigger_retrain_dag(os.environ['AIRFLOW_HOST'], os.environ['RETRAIN_DAG_ID'], os.environ['AIRFLOW_API_TOKEN'])`.
10. Log a summary: `f"Monitor: accuracy={rolling_accuracy}, drift_score={drift_score:.2f}, drifted_features={drifted_features}, retrain_triggered={drift_detected}"`

The monitor task function must be wrapped in a try/except at the top level so that a monitoring failure never blocks the DAG from completing — log the exception and re-raise so Airflow marks the task as failed for visibility, but do not use `on_failure_callback` to suppress.
